# 本地 Anthropic 兼容性探针与能力校验设计

## 背景

Ztest 报告 `01KX8Z86WFWHSZBVJKTTTN6KE3` 的综合分为 60。协议合规、响应结构、缓存字段和流完整性已经通过，但仍存在四类可复现的兼容性问题：

- 请求启用 thinking 后，部分模型没有返回 Anthropic `thinking` 内容块，并触发 `d10_thinking_missing` 硬上限。
- 客户端 system 指令在 Kiro 上游只能表示为普通会话内容，Canary 和 instruction override 的服从率不稳定。
- `tool_choice` 已被请求类型解析，但没有形成完整的强制工具调用语义，结构化输出可能退化为普通文本。
- PDF 的单元测试只验证本地文本提取，没有验证从 HTTP 请求、Kiro wire 到最终回答的完整链路。

S1 Token 注入已经从 `slope=13.19, overhead=6158` 改善到 `slope=1.9, overhead=44`，因此本轮不再调整 Token 统计口径。报告中的 `I'm Kiro`、`Generated by Kiro AI development environment` 等文本在本仓库中不存在，属于上游包装行为，不通过伪造身份或识别检测站点来隐藏。

## 目标

1. 建立可在本地重复运行的 Anthropic 兼容性探针，覆盖 thinking、system/Canary、`tool_choice`、PDF 和并发隔离。
2. 每个生产代码修复必须先由失败测试复现，再以最小改动修复。
3. Thinking 路径基于账号实际模型能力选择原生 reasoning 或兼容回退，不能仅依赖静态模型名。
4. `tool_choice` 的 `auto`、`any`、`tool` 和禁用工具语义在转换结果和最终响应中保持一致。
5. PDF 验证覆盖完整 HTTP/转换/响应链路，并保持文本、文档和图片块的顺序。
6. 模型列表与 reasoning 能力尽量来自 `ListAvailableModels`；查询失败时采用明确、保守且可诊断的回退策略。

## 非目标

- 不识别 Ztest 域名、nonce、探针名称、固定工具名或请求指纹。
- 不伪造 Claude 身份，不重写模型对自身身份的正常回答。
- 不删除或绕过 Kiro 上游 foundational prompt。
- 不承诺固定检测分数，也不把 Kiro 包装描述为 Anthropic 官方直连。
- 不实现扫描版 PDF OCR，也不下载远程 URL 文档。
- 不修改现有 Token 双轨计量、计费、credits 或缓存统计语义。

## 方案选择

采用“本地黑盒探针 + 转换层白盒回归 + 动态能力校验”的组合方案。

只增加白盒单元测试无法发现 HTTP、SSE 和上游事件组合造成的问题；只做黑盒请求则很难定位失败发生在解析、转换、Kiro wire 还是响应渲染。组合方案用同一组行为断言连接两层：白盒测试定位转换不变量，黑盒测试验证客户端实际可见行为。

不直接移植 `jwadow/kiro-gateway` 或 `api-relay-audit` 的代码。这两个项目采用 AGPL-3.0，可作为协议行为和测试分类参考，但本项目使用独立实现。

## 架构

### 本地探针运行器

在仓库内增加独立测试模块或测试工具，不进入生产请求分支。探针运行器接收一个本地服务地址和临时 API Key，构造标准 Anthropic Messages API 请求，并输出结构化结果：

- HTTP 状态和 Anthropic 错误类型。
- content blocks、`stop_reason` 和 usage。
- 流式事件顺序与最终聚合结果。
- 每个探针的通过、失败和诊断原因。

探针只使用运行时生成的随机 Canary，不保存 API Key、完整 system、用户正文、PDF 正文或上游原始凭证。默认测试使用 mock Kiro 上游；真实本地服务测试必须由用户显式提供临时 Key。

### Thinking 能力解析

引入请求级模型能力结果，至少表达：

- 模型是否在当前账号可用。
- 是否接受 Kiro 原生 `additionalModelRequestFields.output_config`。
- 是否观察到原生 reasoning 事件。
- 能力来源是实时模型列表、短期缓存还是保守静态回退。

转换器不得仅凭 `claude-opus-4.8` 字符串就永久关闭文本兼容回退。若账号能力确认支持原生 reasoning，则发送原生字段并期待原生事件；若能力未知，则使用保守策略，确保客户端请求 thinking 时不会静默返回纯文本成功响应。

若上游明确接受原生 reasoning 请求但最终没有 reasoning 事件，响应层先尝试解析既有的严格 `<thinking>...</thinking>` 兼容格式；解析成功则输出标准 Anthropic `thinking` block，解析失败则返回上游协议错误。不得同时宣称请求成功且缺失客户端要求的 thinking 语义，也不得把没有明确 thinking 边界的普通正文伪造成思维链。

### System 与 Canary

由于 Kiro wire 没有等价的高优先级 Anthropic system 角色，本轮不声称能够提供官方直连同等的 system 优先级。探针用于保证本项目自身不引入额外串扰：

- 每个请求使用独立 conversation 和 continuation ID。
- system 文本只出现一次，且保持 block 顺序。
- 不生成额外 assistant 确认语或身份声明。
- 并发 Canary 不得在不同请求之间互串。

如果单请求 system 仍被 Kiro foundational prompt 覆盖，结果标记为上游能力限制，而不是在生产代码里注入更强的身份覆盖提示。

### `tool_choice` 映射

建立内部工具选择枚举，统一解释 Anthropic 请求：

- `auto`：模型可以调用工具或返回文本。
- `any`：必须返回至少一个已声明工具调用。
- `tool`：必须调用指定名称的工具。
- 未提供：保持当前默认行为。

转换阶段验证指定工具是否存在，并把约束传给 Kiro 可表达的原生字段或最小兼容层。响应阶段再次验证不变量：强制调用请求若未产生合法 `tool_use`，返回明确协议错误，不把普通文本伪造成工具调用。

流式与非流式路径共用同一工具选择验证器。`stop_reason=tool_use` 只能由实际输出的 `tool_use` block 推导。

### PDF 全链路

继续支持 base64 文本型 PDF，并保留现有大小、页数、字符数、加密和空文本限制。新增端到端测试验证：

1. HTTP 层接受标准 `document` block。
2. PDF 文本成功提取并按原 content block 顺序进入 Kiro wire。
3. Mock 上游能看到文档 Canary，并在最终 Anthropic 响应中返回它。
4. 任一步失败时返回明确错误，不返回 HTTP 200 空内容。

探针使用仓库内最小静态 PDF fixture 或测试时确定性生成的 PDF，不依赖外部站点。

### 动态模型能力

复用现有 `ListAvailableModels` 客户端，在账号维度建立短期能力缓存。模型请求流程为：

1. 规范化客户端模型名。
2. 查询或读取该账号的可用模型缓存。
3. 若模型不存在，返回明确的 Anthropic `invalid_request_error`，不静默映射到低层模型。
4. 若模型存在，依据返回的规范模型 ID、仓库内经过回归测试的能力表和当前响应是否出现原生 reasoning 事件决定 reasoning 字段；模型列表本身没有声明的能力不得凭显示名称推断。
5. 上游模型列表暂时不可用时，允许配置保守回退；日志记录能力来源，但不记录凭证或请求正文。

公共 `/v1/models` 可以保留兼容别名，但需要区分“服务支持的别名”和“当前账号已确认可用模型”，避免静态列表被理解为每个账号都具有相同能力。

## 测试设计

### Thinking 探针

- `thinking.type=enabled` 且预算大于零时，原生 reasoning 事件转换为带 signature 的 `thinking` block。
- 原生能力未知或缺失事件时，不静默返回只有 text 的成功响应。
- 禁用 thinking 时不产生 thinking block。
- 流式与非流式聚合后的 thinking 内容、顺序和 `stop_reason` 一致。
- 不向已确认原生 reasoning 模型的 system 或 user 正文重复注入 XML。

### System/Canary 探针

- system 要求只返回随机 Canary 时，mock 上游收到且只收到一次该指令。
- 多个 system blocks 保持顺序。
- 32 个并发请求使用不同 Canary，响应不得交叉。
- 相同 `metadata.user_id` 的并发请求仍具有不同 conversation ID。
- 无 system 请求不生成额外 history 文本。

### 工具选择探针

- `auto` 允许纯文本。
- `any` 无工具调用时失败。
- `tool` 指定不存在的名称时在发送上游前返回 400。
- `tool` 指定合法工具时只接受该工具的结构化调用。
- 流式和非流式均满足 tool block 与 `stop_reason` 不变量。
- 普通文本中的 XML 或 JSON 片段不能绕过指定工具校验。

### PDF 探针

- 单页和多页文本 PDF 的 Canary 能完成 HTTP 到响应的往返。
- document 与普通 text 混合时保持顺序。
- 多文档保持原顺序且不会互相覆盖。
- 损坏、加密、无文本、错误媒体类型和超限 PDF 返回明确错误。
- 上游空响应被识别为失败。

### 模型能力探针

- 账号模型列表包含目标模型时正常解析。
- 模型不存在时不降级或冒充。
- 不同账号的模型能力缓存互相隔离。
- 能力缓存过期后重新查询。
- `ListAvailableModels` 临时失败时采用配置的保守回退并产生可诊断日志。

## 错误处理

- 客户端模型、工具选择或文档格式错误返回 Anthropic `invalid_request_error` 和 HTTP 400。
- 上游声明成功但缺少强制 thinking/tool 语义时返回上游协议错误；流式已经开始时发送 Anthropic SSE `error` 后终止。
- 动态模型查询失败与“模型不存在”必须区分，前者可重试，后者直接拒绝。
- 日志只记录探针类别、模型、能力来源、状态码和结构化失败原因，不记录 API Key、Canary 原文、system 正文或 PDF 正文。

## 验证与验收

实施完成后至少运行：

```text
cargo fmt --check
cargo test
cargo check --all-features
cd admin-ui && bun run build
git diff --check
```

验收条件：

- 新增探针在修复前能稳定复现对应失败，在修复后通过。
- Thinking、工具选择、PDF、system 隔离和动态模型能力测试全部通过。
- 现有 Rust 测试和 Admin UI 构建不回归。
- 不包含站点特判、身份伪造、固定 Canary 或 Token 虚报。
- 对 Kiro 上游无法提供的官方 system 优先级和身份一致性给出真实限制说明，不承诺固定 Ztest 分数。

## 风险与回退

- 动态模型查询会增加请求前依赖。通过账号级短期缓存和显式失败分类降低影响。
- Thinking 的保守回退可能增加少量输入 Token。只在能力未知且客户端明确请求 thinking 时启用，并通过探针验证不会重复注入。
- 强制 `tool_choice` 校验可能把以前的自由文本响应改为错误。这是协议语义纠正，应通过版本说明告知调用方。
- 完整 PDF 往返测试可能提高测试时间，fixture 保持最小化且不访问网络。
- 所有生产改动按独立提交组织，若某项上游假设不成立，可单独回退，不影响已稳定的 Token 计量与协议结构修复。
