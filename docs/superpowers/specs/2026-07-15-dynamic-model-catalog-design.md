# RS 动态模型目录设计

## 目标

把公共 `GET /v1/models` 从内置静态数组改为基于 Kiro `ListAvailableModels` 的动态目录，使 NewAPI 等客户端能够发现账号真实拥有的 GPT、Claude 和 Kiro 原生模型，同时保留现有 Claude 模型命名兼容。

本次改动只影响模型发现接口，不改变对话请求体、计费、凭据调度、工具调用或模型本身的响应内容。

## 已确认的现状与根因

- 指定凭据的管理端接口 `GET /api/admin/credentials/:id/models` 已经原样返回上游模型，没有 Claude/GPT 过滤。
- 正式端凭据 `#117` 的实时上游结果包含 `gpt-5.6-sol`、`gpt-5.6-terra`、`gpt-5.6-luna`。
- 公共 `GET /v1/models` 仍调用 `available_models()` 静态数组，因此只返回代码内预先登记的 Claude 与少量 Kiro 原生模型。
- 请求路由已经接受 `claude-opus-4.8` 和 `claude-opus-4-8`，并归一到上游 canonical ID `claude-opus-4.8`；问题只在公共模型列表没有动态发现 GPT。

## 方案选择

采用“按客户端 Key 分组动态汇总 + 五分钟缓存 + 最近成功结果回退”。

不采用以下方案：

- 只查询当前活跃凭据：会漏掉其他订阅账号独有的模型。
- 每次请求都查询全部凭据：首次和持续延迟高，并会放大 `ListAvailableModels` 限流。
- 动态结果失败时回退静态列表：会继续暴露账号实际不可用的模型，与“不要静态列表”的要求冲突。

## 架构边界

### 1. Kiro 动态目录层

新增独立的动态目录组件，负责：

- 按账号分组筛选候选凭据；
- 有界并发查询每个候选凭据的 `ListAvailableModels`；
- 合并、去重模型与上游元数据；
- 缓存每个分组的最近成功目录；
- 把查询结果同步写入现有 `ModelAvailabilityCache`，让后续真实模型请求复用能力判断结果。

该组件不负责 Anthropic/OpenAI 模型别名，也不构造 `/v1/models` 响应。

候选凭据规则：

- 只包含未禁用凭据；
- 客户端 Key 未绑定分组时查询所有未禁用凭据；
- 客户端 Key 绑定分组时，只查询明确包含该分组的凭据，与现有请求调度的严格分组隔离保持一致；
- 临时 429 冷却、RPM 窗口和当前在途数不改变账号的模型权限，因此不会让模型从目录中短暂消失。

### 2. 上游分页层

扩展现有 `ListAvailableModelsResponse` 以读取可选 `nextToken`，并让单凭据模型查询完整翻页：

- 每页继续使用 `maxResults=50`；
- `nextToken` 通过 URL 查询参数编码，不进行字符串直接拼接；
- 没有 `nextToken` 时结束；
- 最多读取 10 页，即单凭据最多 500 个模型；
- 重复出现同一个 `nextToken` 时立即终止并记录协议错误，避免上游异常导致死循环；
- 任一后续页失败时，本次凭据查询视为失败，不把不完整的半份目录合入公共结果。

管理端的单凭据“可用模型”查询也复用同一完整分页函数，因此动态公共目录和管理端看到的结果口径一致。

### 3. 协议展示层

新增独立的公共模型展示转换器，负责把动态上游目录转换为现有 `Model` 响应：

- GPT 模型保持上游原始 ID，不改名、不增加虚构版本；
- Kiro 原生模型保持上游原始 ID；
- Claude canonical ID 与兼容别名同时返回；
- 根据模型前缀填写稳定的 `owned_by`；
- 保留上游提供的展示名，兼容别名使用可读的派生展示名；
- 对结果按 canonical ID 与兼容优先级稳定排序，避免每次刷新顺序抖动。

`GET /v1/models` 改为异步 handler，并读取认证中间件注入的 `KeyContext.group`。不同分组的客户端 Key 会得到不同、且真实可路由的模型目录。

## Claude 命名与 thinking 规则

对于上游返回的 Claude 版本模型，例如 `claude-opus-4.8`：

- 返回 canonical：`claude-opus-4.8`；
- 返回连字符别名：`claude-opus-4-8`；
- 返回 canonical thinking：`claude-opus-4.8-thinking`；
- 返回连字符 thinking：`claude-opus-4-8-thinking`。

如果上游未来返回连字符形式，转换器仍会归一识别并生成同一组结果，不重复输出。

别名生成只匹配明确的 Claude 族与数字版本：

- 支持 `opus`、`sonnet`、`haiku`、`fable`、`mythos`；
- 不改写 GPT、DeepSeek、MiniMax、GLM、Qwen、`auto` 或未知模型；
- 不凭空生成日期快照模型；
- 已经以 `-thinking` 结尾的模型不会再次追加 thinking 后缀。

thinking 是同一上游模型的请求模式兼容入口，不是静态伪造一个新的上游模型。实际请求仍通过现有模型归一化与 thinking 转换逻辑下发。

## 数据合并规则

多个凭据返回同一 `modelId` 时合并成一个 canonical 条目：

- `modelName`：选择第一个非空值；
- `description`：选择第一个非空值；
- `maxInputTokens`：选择所有有效正数中的最大值；
- 空白模型 ID 会被丢弃；
- ID 比较区分实际字符，但 Claude 点号/连字符别名会先归一，避免重复。

公共 `Model.max_tokens` 不会错误使用 `maxInputTokens`。为保持现有 API 契约，它继续使用当前保守的输出上限默认值；上下文窗口仍由既有模型资料与请求逻辑决定。

## 缓存与并发

- 缓存键：客户端账号分组，未分组使用独立的全局键；
- TTL：五分钟，与现有模型可用性缓存一致；
- 错误回退缓存最长保留三十分钟，超过三十分钟不再作为可用目录返回；
- 查询并发：最多 8 个凭据同时请求；
- 同一时刻只有一个目录刷新任务，其他请求等待并复用刷新结果，避免惊群；
- 缓存命中直接返回，不访问 Kiro；
- 成功刷新后原子替换该分组目录。

第一笔缓存未命中的 `/v1/models` 可能比静态列表慢，后续五分钟内为内存读取。模型权限变化最多延迟五分钟体现。

## 错误处理

- 至少一个凭据成功：返回所有成功结果的并集；失败凭据只记录脱敏日志，不让整个目录失败。
- 所有凭据失败但存在该分组三十分钟内的最近成功缓存：返回陈旧缓存并记录告警。
- 最近成功缓存已经超过三十分钟：视为无可用缓存，返回错误。
- 所有凭据失败且没有成功缓存：返回明确的 `502 upstream_model_catalog_error`，不返回空列表，也不回退静态模型。
- 没有任何符合分组且未禁用的凭据：返回 `503 no_available_credentials`。
- 单个模型缺少名称、描述或 Token 信息不会影响该模型出现在目录中。

## `/v1/models` 响应兼容

保持现有外层协议：

```json
{
  "object": "list",
  "data": [
    {
      "id": "gpt-5.6-sol",
      "object": "model",
      "owned_by": "openai"
    }
  ]
}
```

现有客户端 Key 鉴权方式不变。模型映射管理器中的源模型名不会自动加入目录；只有上游真实返回的模型和 Claude 协议兼容别名才会出现。

## 测试策略

### 动态目录单元测试

- 多个凭据结果正确合并、去重；
- GPT 模型保持原名；
- 分组 Key 不能看到其他分组独有模型；
- 无分组 Key 能看到所有未禁用凭据模型；
- 禁用凭据不参与目录；
- 部分凭据失败仍返回成功并集；
- 全部失败时使用最近成功缓存；
- 全部失败且无缓存时返回错误；
- 五分钟内缓存命中不重复查询；
- 并发刷新只触发一轮上游查询。

### 别名转换单元测试

- `claude-opus-4.8` 生成 `4.8`、`4-8` 及各自 thinking；
- 上游输入为 `claude-opus-4-8` 时结果仍然去重；
- `gpt-5.6-sol` 不生成 Claude 风格别名；
- 未知模型原样保留；
- 已有 `-thinking` 不重复追加。

### 路由集成测试

- `/v1/models` 使用 `KeyContext.group` 获取分组目录；
- 响应包含模拟上游 GPT 模型；
- 无 Provider、无凭据和上游全部失败返回稳定错误类型；
- 静态 `available_models()` 不再作为生产目录来源。

## 客户影响

正面变化：

- NewAPI 能自动同步账号真实拥有的 GPT 模型；
- 新模型上线后不需要改 RS 静态代码；
- Claude `4.8` / `4-8` 和 thinking 客户端继续兼容；
- 分组 Key 不会看到无法路由到的其他组模型。

可感知变化：

- 第一次刷新模型列表可能需要等待上游查询；
- 模型列表会随账号订阅、禁用状态和分组真实变化；
- 全部上游失败且没有历史缓存时会明确报错，而不是返回看似正常但不真实的静态列表。
