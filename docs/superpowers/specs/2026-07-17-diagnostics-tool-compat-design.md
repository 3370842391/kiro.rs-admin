# 工具参数兼容与错误现场治理设计

## 目标

降低正式端可安全修复的协议错误，并减少重复错误快照对 SQLite 的膨胀：

1. 对明确无歧义的 `read_file.file_path` → `path` 别名做确定性兼容。
2. 保留 Kiro 流内 `Error`/`Exception` 的安全错误码和截断原因，避免管理端只显示 `Upstream reported error`。
3. 对相同请求指纹、错误类型和响应模式在短时间窗口内合并快照计数，首份记录保留完整现场。

不在本次范围内：接受 `audio/mp3` 作为图片、自动猜测缺失的 `Edit.file_path`/`Bash.command`、长上下文压缩、429 策略调整，以及修改 `kiro.rs`。

## 方案

### 1. 工具参数别名

在工具 Schema 校验前增加一个最小、可证明安全的别名层。仅当：

- 当前输入是 JSON object；
- Schema 的 required 包含 `path`；
- 输入没有 `path`，但有字符串类型的 `file_path`；
- 没有其它冲突字段；

才移动该字段并记录修复路径。其它工具和其它别名保持原有严格校验，避免凭空生成参数。

### 2. 上游错误详情

`AttemptObservation` 已经保存有界的错误消息。流上下文在遇到 `Event::Error`/`Event::Exception` 时不再丢弃 message，而是把错误码和有界原因交给现有 terminal failure / trace protocol-error 管道。对外仍使用稳定错误类型，原因只使用安全截断文本；不保存请求正文或认证信息。

### 3. 快照短窗口合并

在错误快照落库前生成已有的请求指纹，使用 `(fingerprint, error_type, response_mode)` 作为合并键，并设置短窗口（默认 60 秒）。命中时：

- 不重复写入大 payload；
- 增加重复次数和最近时间；
- 保留首份完整请求、响应尾部和诊断信息；
- 管理端列表显示合并次数。

不同请求、不同错误类型、不同响应模式或超过窗口时仍创建独立快照。合并失败不得影响原始错误响应。

## 客户影响

- `read_file` 兼容只修正等价字段，成功调用不改变语义。
- 上游错误详情只改善诊断信息，不改变正常响应。
- 快照合并是内部存储优化，客户无感。
- `audio/mp3` 图片请求仍返回 400，这是正确的客户端协议错误。

## 测试策略

- 单元测试覆盖别名转换的正例、冲突字段和非字符串值。
- 单元测试覆盖 Error/Exception 的错误码和有界 message 进入终止错误及 trace 记录。
- 单元测试覆盖快照首次插入、窗口内合并、窗口外新建、不同错误类型不合并，以及合并失败回退。
- 运行 Rust 全量测试、管理端 Bun 测试、格式检查和现有协议合约测试。
