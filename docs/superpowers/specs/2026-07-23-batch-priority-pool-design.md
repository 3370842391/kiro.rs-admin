# 批量最高优先池设计规格

## 目标

让管理员在凭据列表勾选若干账号后，一次操作将它们设为“最高优先池”。新请求优先使用该池；池内仍按实时并发和首字延迟分流，池中账号都不可用时自动回退到其他账号。

优先级沿用现有语义：数字越小优先级越高，`0` 为最高优先级。

## 当前问题

- 批量编辑仅支持 RPM、分组和来源渠道，不支持优先级。
- 大量账号默认优先级都是 `0`，单独把选中账号设为 `0` 无法形成区别。
- `EndpointMode::Best` 当前强制使用实时并发调度，比较顺序是并发、首字延迟、优先级、历史调度次数；因此优先级只在前两项完全相同时生效。

## 管理端行为

在现有“批量编辑”对话框新增“修改优先级”开关。开启后提供两个互斥模式：

- “指定数值”：输入 `0..=4294967295` 的非负整数，并把所有选中账号设置为该值。
- “最高优先池”：自动建立相对其他账号真正优先的 `0` 级账号池。

数字越小优先级越高；相同数字属于同一优先池。例如 `0` 比 `10` 优先，两个 `0` 会在同一池内实时分流。

最高优先池模式行为：

- 开启后，选中账号组成优先级 `0` 的最高优先池。
- 未选账号的原有优先级层级被稳定压缩到 `1..N`，保留它们原本的相对优先顺序，不持续累加数字。
- 如果选择了全部账号，则全部账号保持优先级 `0`。
- 两种优先级模式都可以与批量 RPM、分组、来源渠道修改同时提交，但同一次请求不能同时指定固定数值和最高优先池。
- 对话框明确提示：最高优先池会优先消耗；池全部不可用、被限流、达到 RPM 或不支持目标模型时才回退。
- 成功提示分别显示选中账号数量和实际被重新编号的账号数量。

不开启“修改优先级”时，所有账号优先级保持不变。

## API 与原子更新

扩展现有接口：

```text
PUT /api/admin/credentials/batch
```

请求新增两个互斥的可选字段：

```json
{
  "ids": [1, 2],
  "priority": 10,
  "promotePriority": false
}
```

或：

```json
{
  "ids": [1, 2],
  "promotePriority": true
}
```

约束：

- `priority` 省略且 `promotePriority` 省略或为 `false` 时不修改优先级。
- `priority` 存在时，将选中账号统一设置为该非负整数，不修改未选账号。
- `priority` 与 `promotePriority: true` 同时出现时返回 HTTP 400。
- 任一种优先级模式都必须先验证所有选中 ID 存在，禁止部分成功。
- RPM、分组、来源渠道和优先级调整在同一次内存锁及持久化操作中完成。
- 持久化失败时恢复所有受影响账号的原状态。
- 响应新增 `priorityAdjusted`，表示包括未选账号重编号在内实际发生优先级变化的账号数。

## 优先级重编号算法

1. 对选中账号赋值 `priority = 0`。
2. 读取未选账号当前优先级，按 `(priority, id)` 稳定排序。
3. 将未选账号现有的相同优先级视为同一层；各层依次映射到 `1..N`。
4. 相同旧优先级的账号继续处于同一新层，避免无意改变原有层级关系。

此算法可重复执行，不会因为多次“设为优先”导致优先级无限增长或溢出。

## “最好模式”调度语义

仅调整 `EndpointMode::Best` 下实时调度的比较顺序：

```text
优先级层级 → 当前 in-flight → 首字 EWMA → 历史调度次数 → 稳定 ID
```

结果：

- 优先级 `0` 的账号池存在可用账号时，新请求只在该池内实时分流。
- 同一优先级池内仍优先选择并发更低、首字更快的账号。
- 最高优先池全部不可用、临时冷却、达到 RPM、被排除或不支持模型时，自动使用下一层。
- 既有会话的 session affinity 保持不变，直到亲和过期或该账号不可用，避免正在进行的对话突然换号。
- 手动端点模式和其他客户端请求协议不变。

## 客户影响

会影响账号消耗顺序：开启操作后，选中账号会承担绝大多数新会话流量，直到不可用或达到限制。

不会改变：

- 客户发送的消息、工具参数和响应内容；
- SSE、首字节握手和缓存协议；
- 模型映射、端点重试链及错误返回格式；
- 已建立会话的短期账号亲和。

如果只选择一个账号且该账号未设置 RPM，它可能承担全部新流量；管理端必须在开关下方显示此风险。

## 测试要求

后端测试：

- 指定 `priority: 0`、`priority: 10` 时只修改选中账号。
- 固定数值与最高优先池同时提交时拒绝且不修改数据。
- 选中账号变为 `0`，未选层级稳定压缩到 `1..N`。
- 重复执行不会继续增加优先级。
- 全选时全部为 `0`。
- 任一 ID 不存在时无修改。
- 持久化失败时完整回滚。
- 最好模式优先选择更高优先级池。
- 同一池内继续按实时并发和首字延迟选择。
- 顶层账号达到 RPM、冷却或禁用时回退下一层。

前端测试：

- 批量对话框展示指定数值/最高优先池两种模式、风险说明和数值方向说明。
- 关闭开关时请求不带 `priority` 和 `promotePriority`。
- 指定数值时支持 `0`、`10` 并拒绝负数、小数、空值和超出 `u32` 的值。
- 开启时请求携带 `promotePriority: true`。
- 仅开启优先池也构成有效批量修改。
- 成功提示展示 `priorityAdjusted`。
