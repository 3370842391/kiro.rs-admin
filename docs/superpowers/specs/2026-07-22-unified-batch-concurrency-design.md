# 批量任务统一并发设计

## 目标

让 GUI 下方的“并发”输入框成为所有批量任务的真实并发上限，避免界面显示的数值与执行器固定默认值 `5` 不一致。

## 范围

统一作用于：登录、导出 JSON、并发登录+提Key、提取 API Key、刷新额度。并发值继续限制为 `1–10`，实际同时运行数为配置值与待处理数量中的较小值。

## 数据流设计

- GUI 每个批量入口启动时调用 `_read_concurrency()`，在确认提示、状态栏和日志中显示该值。
- `run`、`login_and_extract_api_keys`、`login_and_extract_pipeline`、`extract_api_keys`、`refresh_quota` 都显式接收 `concurrency`。
- 提取 Key 的自动补登录阶段和后续提取阶段复用同一个并发值。
- 额度刷新使用 `asyncio.Semaphore` 限制网络任务，并用异步锁保护完成计数和报告聚合。
- 兼容性默认值仍为 `5`，以保留非 GUI 调用方行为。
- Microsoft 登录已有共享浏览器锁，仍可能低于配置并发；这属于协议约束，不改变其串行安全策略。

## 错误处理与进度

- 单账号失败继续处理其余账号，沿用现有报告和事件类型。
- 每个账号只发送一次终态进度事件；并发完成顺序不影响总计数。
- transport 在每个账号任务结束时关闭，即使请求失败也不泄漏连接。

## 测试

- GUI 五个入口都把输入值传给对应协调器调用。
- `login_and_extract_api_keys` 将值同时传给自动登录和提取阶段。
- 提取 Key 与额度刷新通过受控假请求测得最大活跃任务数不超过配置值，配置为 `2` 时确实出现两路并行。
- 现有默认值、错误继续处理、Microsoft 串行锁和报告统计回归测试保持通过。
