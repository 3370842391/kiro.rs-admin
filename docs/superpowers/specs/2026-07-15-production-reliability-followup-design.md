# 生产协议与多模态可靠性收口设计

## 目标

在不移除 New API 一秒 connected/ping、不改变普通文本聊天和 `kiro_native` 原始回复模式的前提下，修复生产错误分析中可稳定复现的结构化输出、图片预算、图片格式、工具 Schema、120 秒流中断和错误诊断问题。

## 已确认边界

- 800 KiB 是 RS 本地选择的图片软目标，不是已证实的 Kiro 全局硬限制。
- 历史图片允许压缩但不允许静默删除；当前轮图片不得有损压缩。
- 不猜测 `file_path`、`path` 或其他工具业务值；非法工具不得交给客户端执行。
- 只有尚未交付正文或工具的请求允许透明重试；已交付部分输出的流不得重放。
- 所有快照继续脱敏，不保存 API Key、Authorization、客户正文和图片 Base64。

## 设计

### 1. 结构化输出归一化

显式 `output_config.format.type=json_schema` 请求先尝试整段 JSON；失败后只提取唯一一个完整 JSON 候选，并使用客户原 Schema 再次严格校验。多候选、缺字段、类型错误、额外字段或超限继续拒绝。归一化成功后只向客户返回压缩后的 JSON 值，不返回 Markdown 围栏或说明文字。

### 2. 图片软预算、硬上限与格式预检

现有 `imageTotalBase64BudgetBytes` 改为软目标；新增可热更新、可持久化、可在管理端编辑的 `imageHardBase64LimitBytes`，默认 8 MiB。普通体超过软目标但低于硬上限时仍可发送，Kiro 明确返回长度阈值后使用激进体重试；普通体超过硬上限而激进体未超时直接发送激进体；两体都超硬上限才本地 400。

所有图片先做 Base64 和 magic bytes 验证。声明格式与真实格式不一致时修正 Kiro format；可解码但上游不兼容的静态图可归一化为标准 PNG。当前轮保持像素无损，不做 JPEG 降质；损坏图片返回带消息/图片索引的本地 400。

### 3. 工具 Schema 原子提交与一次恢复

完整工具块在通过客户 Schema 前保持未提交。首轮非法且没有已交付正文/工具时，丢弃该轮并用公开 Schema 约束加固后的请求重试一次。仅做无歧义类型修复，不补造 required 业务值。第二轮仍非法则返回一次明确 error。

### 4. 流超时与安全重试

底层 reqwest read timeout 不得与应用 watchdog 使用同一 120 秒截止时间；由应用 watchdog 负责稳定分类。零字节/无语义输出的 ReadError 或 IdleTimeout可重试一次；非流式缓冲尚未交付客户内容时也可重试一次；部分流已交付时只发终止 error，不重放。

### 5. 诊断与管理端呈现

本地图片错误记录结构化安全字段；reqwest 错误记录 timeout 分类、error chain 摘要、收到字节和 decoder 缓冲长度。默认日志过滤保留 `kiro_rs=debug`，把 `h2/hyper/reqwest` 降到 info。管理端统计区分 success、recovered warning、client disconnected 和 final error。

## 客户影响

- 普通文本和合法工具请求不增加提示词；结构化输出只在显式 Schema 路径归一化。
- 图片长会话更少本地误拒绝；历史截图可能在真实阈值重试时更模糊，当前轮像素保持无损。
- 仅异常输出、非法工具或零输出中断会多一次上游调用，因此可能增加该失败请求的延迟和用量。
- 已交付部分内容的请求不自动重试，避免重复正文和重复工具执行。

## 验收

- 252 个结构化历史形状全部可恢复，212 个首轮完成、40 个第二轮完成。
- 944,788 字节图片普通体不再因 800 KiB 软目标本地 400。
- 两种图片体都超过硬上限时才本地拒绝，且当前轮图片像素不变。
- 非法工具不会执行；可恢复样本最多一次重试并只交付一个合法工具块。
- 零输出 120 秒中断最多重试一次；部分输出不中途重放。
- 一秒 connected/ping 回归测试继续通过。
