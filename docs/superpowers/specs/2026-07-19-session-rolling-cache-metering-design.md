# 会话滚动缓存计量优化设计

## 1. 背景

RS 的 Anthropic 缓存字段来自本地前缀哈希计量，不是 Kiro 上游返回的真实 Prompt Cache。该模块只拆分：

```text
input_tokens + cache_creation_input_tokens + cache_read_input_tokens
```

三项总量，不减少 Kiro 上游 token、credits 或响应延迟，但会影响下游按 Anthropic 口径计费时的价格拆分。

生产环境在 2026-07-18 重新开启自动缓存后，配置为：

```text
cacheMeteringEnabled=true
cacheAutoWithoutControl=true
cacheDefaultTtlSecs=1800
cacheCapacity=4096
cacheHitRateMinPct=0
cacheHitRateMaxPct=95
```

以配置修改时间为边界，并按 `request_start = ts - durationMs` 排除切换前启动的在途请求后，约 12.2 分钟内观察到：

| 指标 | 结果 |
|---|---:|
| 缓存请求 | 3,266 |
| 被检查的历史前缀段 | 224,724 |
| 命中段 | 58,738 |
| 未命中并重新登记的段 | 165,986 |
| 段命中率 | 约 26% |
| 未命中登记速度 | 约 13,610 项/分钟 |
| 4096 容量理论周转时间 | 约 18.1 秒 |
| 上报 cache_creation | 83,856,513 tokens |
| 上报 cache_read | 34,112,776 tokens |
| creation 占缓存 token | 71.08% |

日志中同时存在 2,223、1,113 和 1,290 条消息的三个活跃长会话。现有算法为几乎每个历史消息边界建立一条递增前缀，三个会话合计至少需要 4,626 条记录，已经超过全局 4,096 容量。TTL 虽然配置为 30 分钟，条目却通常在几十秒内被 LRU 容量淘汰。

## 2. 目标

1. 保留 `cacheAutoWithoutControl=true` 的模拟读取能力。
2. 让连续多轮会话主要上报 `cache_read`，只把新增长度上报为 `cache_creation`。
3. 单个超长会话不能一次登记数千条缓存记录。
4. 30 分钟 TTL 在当前生产请求规模下真正发挥作用，不再被 4096 容量提前淘汰。
5. 不修改响应正文、工具调用、SSE 顺序、首字节握手和上游请求内容。
6. 不通过固定比例或无条件伪造命中来隐藏冷启动；真正的新会话仍应产生 creation。
7. 保持三项 input token 总量守恒。
8. 提供独立运行时开关，管理员可以无重启启用滚动窗口或恢复旧的全历史前缀算法。

## 3. 非目标

- 不实现 Kiro 上游真实 Prompt Cache。
- 不承诺降低 Kiro credits、上游成本或上游延迟。
- 不把所有请求强制整形为固定 90% 或 95% read。
- 不把冷启动的 `cache_creation` 强制改为 0。
- 不修改已经落盘的历史 usage 记录。
- 不改变客户请求正文、图片、工具参数或助手输出。

## 4. 方案比较

### 4.1 仅扩大容量

把容量从 4,096 调整为 65,536，可以暂时延后淘汰，但现有算法仍会在每次长对话请求中查询和登记上千个前缀。按实测未命中写入速度静态计算，65,536 也只能覆盖约 4.8 分钟，而且会扩大 LRU 排序和 DEBUG 日志开销。

该方案只能作为配套措施，不能单独采用。

### 4.2 固定比例提高 cache_read

在 usage 输出阶段直接把 creation 转移到 read，可以立即改善账单数字，但它与会话是否存在稳定前缀无关，会产生冷请求命中、跨会话命中以及计量检测异常。

该方案不采用。

### 4.3 最近前缀滚动窗口

每个请求只查询和登记最近少量、最有可能在下一轮复用的递增前缀。连续对话的上一轮最深前缀仍在窗口内，因此下一轮可以读取已有历史，只把新增的 user、assistant 或 tool 消息计算为 creation。

该方案同时降低缓存条目增长、查询成本、淘汰压力和日志体积，是本设计采用的方案。

## 5. 核心算法

### 5.1 完整前缀计算与候选窗口分离

请求解析仍按当前顺序计算结构化前缀：

```text
tools -> system -> messages
```

算法仍需要得到完整 prompt 的估算 token 和最深可缓存前缀，以保证 usage 总量正确；但 CacheMeter 不再 lookup/record 全部历史边界。

从完整前缀链末尾选择最近 `rolling_prefix_limit` 个候选，默认值为 8：

```text
all_segments = [s0, s1, ..., s2199, s2200, s2201]
cache_candidates = [s2194, ..., s2201]
```

因此 2,223 条消息的请求最多登记 8 条记录，而不是约 2,223 条。

### 5.2 为什么最近 8 个前缀仍能稳定读取

正常连续对话每轮通常增加两条历史消息：上一轮 user 和 assistant。下一轮请求中，上一轮最深缓存前缀一般位于当前前缀链倒数第 2 至第 4 个位置，仍在最近 8 个候选内。

计算规则保持：

```text
cache_read = 最深命中候选的累计 tokens
cache_creation = 当前最深可缓存前缀 tokens - cache_read
input = 可见总 input - cache_read - cache_creation
```

预期行为：

- 新会话首轮：read=0，稳定前缀进入 creation。
- 正常下一轮：绝大部分旧历史进入 read，仅新增历史进入 creation。
- 工具调用多阶段续轮：上一轮附近的多个前缀仍在窗口内，可以容忍一次增加多条 tool_use/tool_result 消息。
- 客户端一次性补入超过 8 个新历史边界时：允许回退为部分命中或重新创建，不伪造 read。

### 5.3 短请求

当前缀总数不超过 8 时全部参与 lookup/record。没有稳定前缀时继续返回普通 input，不产生空缓存记录。

### 5.4 会话隔离

保留现有隔离种子规则：优先使用 `metadata.user_id` 中的 `_session_...`，否则使用客户端 Key id。滚动窗口只减少被登记的前缀数量，不删除隔离种子，也不允许不同隔离种子直接复用完整前缀 hash。

本轮不引入明文 session id 落盘。日志和持久化文件继续只保存不可逆 hash、token 数和时间戳。

### 5.5 运行时开关

新增独立开关：

```text
cacheRollingPrefixEnabled=true
```

- 开启：使用 v2 最近前缀滚动窗口，受 `cacheRollingPrefixLimit` 限制。
- 关闭：恢复当前 v1 全历史前缀 lookup/record 行为，用于紧急回退和对比测试。
- 保存后立即生效，不需要重启，不自动清空缓存和 usage 历史。
- 该开关独立于 `cacheMeteringEnabled` 和 `cacheAutoWithoutControl`；关闭滚动窗口不等于关闭缓存计量。
- v1/v2 使用不同内部哈希命名空间，切换时不会把另一种算法的条目误判为命中；切换后的首轮允许正常冷创建。
- 管理端关闭开关时显示警告：长对话可能再次产生大量 creation 并快速占满容量。

## 6. 容量与淘汰策略

### 6.1 默认参数

新增结构化配置：

```text
cacheRollingPrefixEnabled=true
cacheRollingPrefixLimit=8
```

允许范围为 2–64。生产推荐：

```text
cacheCapacity=65536
cacheRollingPrefixEnabled=true
cacheRollingPrefixLimit=8
cacheDefaultTtlSecs=1800
```

每次请求最多新增 8 条；正常 warm 续轮通常只新增 1–3 条。按当前每分钟约 300 个请求估算，65,536 容量可以覆盖 30 分钟量级的滚动前缀，而不是在 18 秒左右周转。

### 6.2 批量 LRU 淘汰

现有实现每次超过容量都会收集并排序全部条目。容量扩大后，不应在每个新请求上排序约 65,536 条记录。

改为批量淘汰：

- 未达到 capacity 时不排序。
- 达到 capacity 后，一次淘汰到 `capacity * 95%`。
- 淘汰目标仍按 `last_hit_at` 从旧到新选择。
- 过期条目优先清理，不计入 5% 活跃条目淘汰目标。

这样每释放约 3,277 个槽位才需要下一次全量选择，避免持续满载时每请求排序。

### 6.3 TTL

TTL 规则保持不变：显式支持的客户端 TTL 优先，否则使用管理端默认值。延长 TTL 不能替代滚动窗口和合理容量；只有条目没有提前被容量淘汰时，30 分钟 TTL 才有实际意义。

## 7. usage 与命中率整形

滚动窗口先产生基于真实本地前缀状态的原始 `cache_read/cache_creation`，再沿用现有命中率上限逻辑。

生产当前：

```text
cacheHitRateMinPct=0
cacheHitRateMaxPct=95
```

含义保持为：

- min=0：不把冷请求伪造成命中。
- max=95：必要时压低异常高的 read 占比。
- 不允许 max 逻辑反向增加 read。
- 三项 input token 拆分前后总量必须完全一致。

不新增“强制 read 比例”开关。

## 8. 持久化与升级

缓存文件仍使用 `cache_metering.json`，现有 entry 字段无需增加客户或会话明文。滚动窗口只改变未来 lookup/record 的候选集合，旧 JSON 可以继续加载。

为避免旧算法产生的全历史前缀被误认为新算法热缓存，滚动模式的前缀哈希种子加入内部算法版本：

```text
cache-meter:v2
```

升级影响：

- 部署后的每个会话第一次请求会有一次正常冷创建。
- 旧 v1 条目不会命中，会按 TTL 到期或容量策略被淘汰。
- 关闭滚动开关时使用 v1 命名空间；重新开启后返回 v2 命名空间，两边条目可以在 TTL 内共存但不会交叉命中。
- 无需删除 usage 历史。
- 无需在升级时强制清空文件，避免启动路径加入破坏性操作。

## 9. 可观测性

当前 DEBUG 会为一个 2,000 段请求输出数千个 hash，日志体积过大。改为单行摘要：

```text
CacheMeter: all_segments=2223 candidates=8 hits=6 misses=2 deepest_hit=5 read=143021 creation=486 ttl=1800
```

默认不输出：

- prompt 正文；
- system 内容；
- tool 参数；
- session id；
- 全量 hash 列表。

管理端缓存状态增加运行期计数：

```text
rollingPrefixLimit
segmentLookups
segmentHits
segmentMisses
evictions
expiredEntriesRemoved
```

这些计数用于排障，不写入客户 usage，也不参与计费。

## 10. 管理端

缓存策略界面增加“最近前缀滚动缓存”开关以及“每请求滚动前缀数”，开关默认开启、数量默认 8，并显示说明：

> 仅保留最近的可复用历史前缀。数值越大越能容忍分支或工具多阶段续轮，但会增加内存与淘汰压力。一般保持 8。

关闭开关后隐藏或禁用数量输入，并展示：

> 已恢复旧的全历史前缀算法。超长对话可能一次写入数千条缓存记录，建议只用于临时对比或紧急回退。

容量字段推荐值调整为 65,536，但升级时不静默覆盖用户已有配置。管理员保存后运行时生效并持久化到 `config.json`。

缓存状态显示：

- 有效条目/最大容量；
- 最近运行期段命中率；
- 累计淘汰数量；
- 滚动前缀限制；
- 滚动算法是否启用；
- TTL 和自动缓存开关。

## 11. 错误处理

- `cacheRollingPrefixLimit` 不在 2–64：管理 API 返回 HTTP 400，运行时策略不变。
- 配置持久化失败：不更新运行时策略，返回 HTTP 500。
- 缓存 JSON 损坏：记录 warning，从空缓存启动，不阻断对话。
- 计量计算异常：回退为全部普通 input，不中断对话、不伪造 read。
- 运行时计数溢出：使用饱和累加或无损重置，不影响请求。

## 12. 客户影响

### 12.1 正面影响

- 连续长对话的旧历史更稳定地显示为 cache_read。
- `cache_creation` 从反复全量创建变为主要记录每轮新增部分。
- 使用按缓存折扣计费的下游客户通常会降低实付成本。
- 缓存 lookup、record、LRU 和 DEBUG 日志开销下降。

### 12.2 必须明确的商业影响

- Kiro 上游真实成本不会同步下降。
- 下游账单降低可能压缩运营毛利。
- 首轮、真正的新会话或大幅重写历史仍会产生 creation。
- 部署 v2 后会有一次冷启动窗口。

### 12.3 不受影响

- 对话正文；
- 工具调用参数和 tool_use/tool_result 配对；
- 流式与非流式协议；
- 首字节握手；
- 图片和 PDF 处理；
- 模型映射和凭据选择；
- Kiro 上游 token/credits。

## 13. 测试设计

### 13.1 单元测试

1. 2,223 条消息最多产生 8 个 lookup/record 候选。
2. 同一会话下一轮增加两条消息后，上一轮最深前缀命中，creation 只覆盖新增部分。
3. 新 session 的首轮 read 为 0。
4. 不同 session 的相同正文不发生完整前缀串用。
5. 最近 8 个候选能覆盖常见 tool_use/tool_result 续轮。
6. 一次增加超过 8 个历史边界时安全回退，不产生负数或总量漂移。
7. `input + creation + read` 在整形前后均守恒。
8. 批量淘汰优先删除过期项，并下降到 95% 容量目标。
9. 非法滚动前缀配置被拒绝。
10. v1 持久化文件可加载，但不会与 v2 hash 命中。
11. 运行时关闭滚动开关后恢复全历史候选，重新开启后立即恢复最多 8 个候选。
12. 开关切换不清空缓存、不修改历史 usage，也不影响正在处理的请求所持有的策略快照。

### 13.2 回归测试

- `cacheAutoWithoutControl=false` 且请求无 cache_control 时仍为全 input。
- 缓存总开关关闭时不 lookup、不 record。
- 滚动开关关闭时保留旧算法行为，但不影响缓存总开关和自动缓存开关的语义。
- 显式 TTL 和默认 TTL 规则保持现状。
- 命中率上限不能制造冷启动 read。
- direct/NewAPI、流式/非流式 usage 总量一致。
- SSE、工具调用和首字节相关测试保持通过。

### 13.3 负载回放

构造与生产日志相近的混合负载：

- 一个 2,200+ 消息连续会话；
- 两个 1,000+ 消息连续会话；
- 每分钟约 300 个短请求；
- TTL 30 分钟；
- capacity 65,536。

对比 v1/v2：

| 验收指标 | v2 目标 |
|---|---:|
| 单请求候选条目 | 不超过 8 |
| 长会话 warm 轮次最深前缀 | 应命中 |
| 因容量导致的理论保留时间 | 达到 30 分钟量级 |
| 连续长会话缓存 read 占比 | 通常 80%–95% |
| creation | 主要对应新增历史 |
| DEBUG 单请求日志 | 常数大小，不随消息数线性增长 |

聚合 read 比例不能对大量一次性冷请求作硬保证；验收以连续多轮会话为主，不能为了达到固定百分比修改冷启动事实。

## 14. 发布与验证

1. 本地单元测试和回归测试通过。
2. 部署到隔离公网 8991，保留生产相同 TTL 和滚动参数。
3. 用长会话回放确认缓存条目增长、read/creation 和日志摘要。
4. 观察 8991 至少 30 分钟，确认无缓存容量快速周转、无对话中断。
5. 再部署生产；生产部署后预期出现一次 v2 冷创建。
6. 生产观察首个 30 分钟窗口，按 key/model 对比 creation-only、read-positive、淘汰数与 entry 年龄。
7. 若出现异常，可关闭 `cacheRollingPrefixEnabled` 立即恢复 v1；也可关闭自动缓存或缓存总开关。任何回退都不修改请求正文。

## 15. 设计结论

采用“可运行时开关的最近 8 个前缀滚动窗口 + 65,536 全局容量 + 95% 批量 LRU”的组合。默认启用滚动算法，管理员可以无重启切回旧算法。该方案从源头消除单个长对话一次写入数千项的容量放大，不依赖固定比例伪造 read，并保持对话、工具、SSE 和上游调用行为不变。
