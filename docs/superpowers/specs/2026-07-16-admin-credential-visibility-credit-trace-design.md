# 管理端账号完整展示、可用积分与 429 链路设计

## 1. 背景与目标

当前账号管理页存在三个可观测性问题：

1. 账号卡片的身份徽章容器使用 `overflow-hidden`，Auth Region、API Region、分组、来源渠道和端点等信息在宽度不足时被直接裁掉。
2. 后端和前端已经持有单账号余额，但顶部状态栏没有汇总当前启用账号的可用积分，管理员只能逐个查看。
3. 凭据失败弹窗把一次请求中的失败跳展平成多条记录，隐藏了后续端点/账号尝试；同时把客户端 Key 名当成无标签徽章，并把所有最终成功都描述成“由其他凭据成功”，无法判断 429 是否已重试、切换了哪个端点以及是否换了账号。

本次目标是只改管理端 UI 和前端派生统计，让账号信息、积分覆盖率和请求级 429 恢复链路直接可见。后端已记录完整 `TraceRecord.attempts`，生产样本也确认 IDE 默认链会按 `ide → runtime → codewhisperer → amazonq` 尝试，因此本次不改重试、调度或端点链行为。

## 2. 范围

### 2.1 本次包含

- 账号卡片的身份徽章自动换行并完整显示。
- 顶部 RPM 状态栏增加启用账号可用积分及余额覆盖率。
- 凭据失败弹窗按外部请求分组，展示该请求的完整尝试链和准确的恢复结论。
- 为积分派生逻辑、429 恢复判定和 UI 接线补充前端测试。

### 2.2 本次不包含

- 不修改后端 429 重试次数、冷却、账号调度或端点顺序。
- 不增加余额查询接口，不主动轮询上游。
- 不修改聊天请求、响应正文、SSE、首字节、Token 统计或计费。
- 不扩大 trace/error snippet 的采集范围，也不改变日志保留策略。

## 3. 账号卡片完整展示

### 3.1 已确认根因

`admin-ui/src/components/credential-card.tsx` 中账号名称下方的徽章容器当前为：

```tsx
<div className="mt-1 flex min-w-0 items-center gap-1 overflow-hidden [&>*]:shrink-0">
```

子项禁止收缩且父容器隐藏溢出，宽度不足时后续徽章必然消失。

### 3.2 布局规则

- 账号名称/邮箱继续保持单行截断，避免极长身份文本无限撑宽。
- 徽章容器移除 `overflow-hidden`，增加 `flex-wrap`，使用独立的横向和纵向间距。
- 每个徽章仍保持自身不收缩，换到下一行后完整显示。
- 账号行高度随徽章行数自然增加；拖拽、选择框和右侧指标仍与整行垂直居中。
- 不增加横向滚动，不使用“+N”折叠，也不依赖悬浮提示才能看到完整信息。

## 4. 可用积分汇总

### 4.1 数据来源与优先级

前端从 `CredentialStatusItem.balance` 获得后端最近一次余额缓存，从 `balanceMap` 获得管理员在当前页面手动刷新后的更新值。每个账号按以下优先级选取余额：

1. `balanceMap.get(credential.id)`；
2. `credential.balance`；
3. 无余额数据。

此优先级与账号卡片现有展示一致，确保顶部汇总不会落后于刚刚手动刷新的单账号余额。后端余额缓存仍为 5 分钟，本次不引入额外上游请求或 Token 消耗。

### 4.2 精确定义

新增纯前端 helper，输入当前全部 `credentials` 和 `balanceMap`，输出：

- `enabledCount`：`disabled === false` 的账号数量。
- `observedCount`：启用账号中存在余额且 `remaining` 是有限数值的账号数量；零余额和负余额也算“已统计”，因为它们是有效观测结果。
- `availableCredits`：启用账号中所有 `remaining > 0` 的有限数值之和；零值、负值、`NaN` 和无穷值不加入总额。

额外规则：

- 禁用账号完全排除在分子、分母和总额之外。
- 仅处于 429 临时冷却、但没有被禁用的账号继续计入，因为其积分没有因此消失。
- `observedCount === 0 && enabledCount > 0` 时主值显示“待查询”，避免把未知误报为 `$0`。
- `observedCount > 0` 且正余额之和为零时显示 `$0.00`，表示已观测账号确实没有可用正余额。
- `enabledCount === 0` 时显示 `$0.00`，详情显示“无启用账号”。
- 有启用账号时详情显示“已统计 X/Y 个启用账号”；覆盖率不满时仍展示已知总额，但明确保留覆盖率，避免把部分数据理解成全量。

金额使用固定 USD 货币格式和千分位，例如：

```text
$118,320.45
已统计 12/15 个启用账号
```

### 4.3 展示位置

`RpmStatusBar` 增加一个“可用积分”状态项，并接收积分汇总对象。响应式网格允许状态项自然换行，不压缩现有 RPM、容量、满载账号和进行中请求信息。

## 5. 429 与失败尝试链

### 5.1 请求级分组

弹窗仍使用 `failedAttemptCredentialId` 查询最近最多 50 个“该凭据至少失败过一跳”的外部请求，但不再把失败跳展平。每个 `TraceRecord` 渲染为一个请求卡片，并展示该请求的全部 `attempts`，包括其他账号的后续尝试和最终成功跳。

尝试严格按 `attempt` 升序展示；若后端意外返回乱序数据，前端在副本上排序，不原地修改查询缓存。

请求头明确区分字段：

```text
客户端 Key：newapi
模型：claude-opus-4-8
尝试 4 次（含 3 次重试）
```

`keyName` 缺失时显示 `客户端 Key：#<keyId>`，不再让 `newapi` 看起来像端点名称。

### 5.2 每跳展示

每个尝试至少展示：

- 跳序号；
- 账号（邮箱可用时显示邮箱，否则显示 `#credentialId`）；
- `endpoint`，缺失时显示“未知端点”；
- HTTP 状态，网络层无状态码时显示“网络错误”；
- `durationMs`；
- outcome 的中文标签；
- 已有的截断错误摘要 `errorSnippet`（存在时）。

紧凑链路摘要采用可换行的箭头形式，例如：

```text
#202 / ide 429 → #202 / runtime 429 → #203 / ide 429 → #203 / runtime 200
```

详细行与摘要使用同一排序结果，避免二者顺序不一致。

### 5.3 恢复结论

恢复结论相对于当前打开失败日志的 `credentialId` 判定：

1. `finalStatus === "success"` 且 `finalCredentialId !== credentialId`：显示“切换其他账号后成功”。
2. `finalStatus === "success"` 且 `finalCredentialId === credentialId`：
   - 最终成功跳的端点与该账号先前失败跳的端点不同，显示“同账号切换端点后成功”；
   - 端点相同或无法证明发生端点变化，显示“同账号重试后成功”。
3. `finalStatus === "interrupted"`：显示“流式响应中断”，并保留已发送字节等现有请求级信息（若界面已有）。
4. 其他非成功终态：显示“最终失败”。
5. `attempts` 为空：显示“请求未到达上游”，不伪造重试链。

该规则避免把同账号换端点成功误报为“其他凭据成功”，也避免在同端点重试成功时误报为“切换端点”。

## 6. 前端数据流

```text
CredentialStatusResponse.credentials
  ├─ credential.balance ─┐
  └─ balanceMap（优先） ─┴─> 可用积分纯函数 ─> RpmStatusBar

useTraces({ failedAttemptCredentialId })
  └─ TraceRecord[]
       └─ attempts 排序 + 恢复判定纯函数 ─> 请求卡片 ─> 完整尝试链
```

纯函数与渲染组件分离，便于覆盖边界情况，也避免在 JSX 中重复实现统计和恢复判断。

## 7. 空状态与异常数据

- 余额尚未查询：显示“待查询”和 `0/Y` 覆盖率，不显示虚假的零积分。
- 部分余额未知：显示已知正余额总额和真实覆盖率。
- 非有限余额：视为未知，不污染总额，也不计入覆盖数。
- 失败记录为空：保留“近期无失败或 trace 未开启”的空状态。
- 尝试记录为空：显示“请求未到达上游”。
- 端点、邮箱、HTTP 或摘要缺失：使用明确占位，不隐藏整条尝试。
- 错误摘要继续使用可滚动、自动换行区域，避免超长上游错误撑破弹窗。

## 8. 测试策略

实施阶段遵循 TDD，先增加失败测试再修改代码：

1. `credential-summary` 单元测试：启用/禁用、正/零/负余额、未知和非有限余额、`balanceMap` 覆盖后端缓存、冷却账号仍计入、无启用账号。
2. 429 派生逻辑单元测试：同账号换端点成功、同账号同端点重试成功、换账号成功、最终失败、流中断、空 attempts 和乱序 attempts。
3. UI 合约测试：账号徽章容器允许换行且不再隐藏；状态栏接收并展示积分与覆盖率；失败弹窗按请求渲染完整 attempts，并明确标注客户端 Key 和 endpoint。
4. 回归验证：`bun test` 与 `bun run build` 必须通过。

不要求 Rust 后端测试，因为本次设计不修改 Rust、数据库或 API；若实施时发现必须改变接口，应暂停并重新评审范围。

## 9. 客户影响

### 9.1 可见变化

- 徽章较多的账号行会变高，但所有信息可直接看到，列表不再静默裁剪。
- 顶部会显示当前已知的启用账号可用积分和统计覆盖率；余额最多可能沿用后端 5 分钟缓存，手动刷新后立即采用当前页面的新值。
- 失败弹窗会显示完整 429 重试和端点/账号切换过程，原先错误的“其他凭据成功”文案会被纠正。

### 9.2 不受影响

- 客户聊天内容、上下文连续性、工具调用和图片处理不变。
- 1 秒首字节/SSE 心跳逻辑不变。
- 上游请求次数、429 重试策略、端点链和账号切换策略不变。
- 输入/输出 Token、缓存 Token 和客户计费不变。
- 不增加余额查询流量；仅汇总已有数据。

## 10. 验收标准

- 任意数量徽章均能换行完整显示，邮箱仍为单行。
- 积分总额严格排除禁用账号和非正余额，并准确显示已统计/启用账号数。
- 未取得任何启用账号余额时显示“待查询”，已知全为零时显示 `$0.00`。
- 429 弹窗按请求展示所有尝试，能直接看出每跳账号、端点、HTTP、耗时和结果。
- 同账号换端点成功、同账号同端点重试成功、换账号成功三类文案互不混淆。
- 前端测试与生产构建全部通过。
