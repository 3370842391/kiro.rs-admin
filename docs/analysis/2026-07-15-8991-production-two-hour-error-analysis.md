# 8991 图片预算与生产端两小时错误分析

> 分析日期：2026-07-15
>
> 性质：只读诊断与修复建议；本轮未修改代码、配置、容器或线上数据
>
> 数据安全：报告不包含 API Key、Authorization、客户正文、工具参数值、凭据编号或完整请求体

## 1. 结论摘要

1. 8991 上的图片 400 是 **RS 本地图片预算预检主动拒绝**，不是 Kiro 上游返回。该请求有 1,811 条消息和 12 张历史图片；普通压缩后图片 Base64 总量仍为 944,788 字节，超过当前 819,200 字节预算 125,588 字节，因此在上游调用前结束，`total_attempts=0`。
2. 当前图片预算流程存在一个可修复的控制流问题：普通压缩稍微超限时直接返回 400，代码没有先尝试已经配置好的 `960px / JPEG 60` 激进历史压缩体。推荐先修这个流程，再考虑调大预算。
3. 生产端固定两小时窗口共有 14,146 次请求，13,753 次成功，原始成功率 97.22%。393 次非成功中，有 45 次是客户端主动断开，252 次集中在同一个客户端 Key 的 Claude CLI 标题 JSON 请求；不能把它们全部当成 RS 服务故障。
4. 最大的可修复项是 `output_config.format`：252 个失败样本中，212 个首轮响应已经包含唯一且 Schema 合法的 JSON；剩余 40 个在第二轮全部包含合法 JSON。RS 只是因为 JSON 外面有 Markdown 围栏或少量说明文字而拒绝。使用“唯一 JSON 提取后再次严格校验 Schema”可让这 252 个历史样本全部成功，并且只影响显式结构化输出请求。
5. 25 次工具 Schema 失败也值得修复。现有代码虽然有一次重试设计，但生产 trace 均显示 `total_attempts=1`，说明这些请求在满足重试条件前就被判定为已经提交语义输出，恢复路径没有真正执行。
6. 17 次 `error decoding response body` 和 1 次流空闲超时都集中在约 120 秒，且上游 HTTP 已为 200，分布在多个凭据上。现有 reqwest `read_timeout=120s` 与应用层流 watchdog `120s` 发生竞态，导致大量超时被记成模糊的解码错误。该项可改善诊断，并可对“尚未向客户交付正文”的请求做一次安全重试。
7. 生产窗口内没有最终失败的 429。错误快照中另有 80 条已恢复 warning，均对应最终成功请求，不应计入客户失败率。

## 3. 8991 图片预算 400

### 3.1 已确认事实

| 字段 | 值 |
|---|---:|
| 模型 | `claude-opus-4-8` |
| 请求模式 | 流式、`detection` |
| 消息数 | 1,811 |
| 图片数 | 12 |
| 原始图片 Base64 总量 | 约 1,993 KiB |
| 普通历史压缩后 | 944,788 bytes |
| 当前预算 | 819,200 bytes（800 KiB） |
| 超出 | 125,588 bytes（约 15.3%） |
| 上游尝试数 | 0 |
| 本地返回耗时 | 606 ms |

12 张图片都位于较早的历史消息中，最后一张距离会话结尾仍约 566 条消息。这说明问题不是当前轮图片过大，而是极长会话持续携带早期视觉上下文。

测试端数据库在相邻时间只记录到一次对应的图片预算 HTTP 400。界面中多次显示同一错误文案，不代表服务器实际收到了三次完全相同的失败请求；后续三次请求中有两次成功，另一次是独立的 `stream_read_error`。客户端是否复用了上一条错误展示，需要在客户端侧单独核对。

### 3.2 根因

当前策略位于 `src/kiro/image_budget.rs`：

- 默认预算为 819,200 字节；
- 普通历史压缩为最长边 1,280、JPEG 质量 72；
- 激进压缩为最长边 960、JPEG 质量 60；
- 当前轮图片不压缩，只处理历史图片。

问题出在 `prepare_kiro_bodies` 的执行顺序：普通压缩通过 `?` 直接传播 `ImageBudgetError::Exceeded`，因此普通压缩超限时函数立即退出；后面的激进压缩体根本没有机会生成。激进体目前主要用于“本地已通过、但 Kiro 返回请求体过大”后的重试，而不能挽救本地预检稍微超限的请求。

### 3.3 推荐修复

#### P0：先尝试激进历史压缩，再决定是否 400

推荐流程：

1. 生成普通历史压缩体。
2. 普通体低于预算时保持现有行为。
3. 普通体超过预算时，不立即返回 400；从原始请求生成激进历史压缩体。
4. 激进体低于预算时，把激进体作为本次主请求发送。
5. 两种压缩都超限时才返回 400，并在安全诊断中记录图片数、历史/当前轮数量、压缩前后字节数和预算。
6. 当前轮图片继续保持不变。

可修复性：**高**。该案例全部是历史图片，激进压缩有较大机会降到预算以下，但必须用脱敏 fixture 验证，不能在缺少原始图片时保证一定通过。

客户影响：

- 仅超预算的多图长会话受到影响；普通文本、单图和预算内请求不变。
- 较早的历史截图会更模糊，当前轮图片质量不变。
- 图片预处理增加少量 CPU 和通常几十至几百毫秒延迟；当前失败请求本来已经花费约 606 ms 做普通压缩。
- 不增加隐藏提示词和正常请求 Token；若成功送达上游，原本失败的请求会产生正常上游用量。

#### 临时配置：把预算提高到 1 MiB

该案例压缩后为 944,788 字节，1 MiB（1,048,576 字节）可以放行。可在 8991 临时验证，但不建议把“调大预算”当成唯一修复。

客户影响：历史图片质量比激进压缩更好，但更大的请求更可能被 Kiro 真正的请求体阈值拒绝，届时客户会等待更久才失败，也会增加出口带宽。

#### 不推荐：静默删除最旧图片

它能让超长会话继续，但模型会失去早期视觉上下文，可能错误理解后续“按之前截图修改”等指令。若未来实现，只能作为明确可配置的降级策略，并在响应/日志中暴露已淘汰图片数量，不能默认静默启用。

## 5. 分项根因、修复建议与客户影响

### 5.1 P0：结构化输出被过度严格拒绝（252 次）

#### 证据

- 252 次全部来自同一个客户端 Key。
- User-Agent 为 `claude-cli/2.1.92 (external, cli)`。
- 模型全部为 `claude-haiku-4-5`，请求均为流式 `detection`。
- Schema 全部是单字段对象：必须包含字符串 `title`，不允许额外字段。
- 每次都执行了两次上游尝试，平均耗时约 2.83 秒，最终仍失败。
- 对 252 份脱敏上游响应做结构分析：
  - 212 份首轮输出已经包含且只包含一个满足 Schema 的 JSON 对象；
  - 其余 40 份第二轮都包含且只包含一个满足 Schema 的 JSON 对象；
  - 第二轮中 251 份是完整 Markdown JSON 围栏，1 份是围栏/JSON 外又附带说明文字；
  - 没有发现多 JSON 值歧义。

#### 根因

`src/anthropic/structured_output.rs` 目前只对整段文本直接执行 `serde_json::from_str`。即使输出中只有一个 Schema 合法的 JSON，只要外面有 ```json 围栏或一句说明，就会被判为 `InvalidJson`。

仓库的 `src/anthropic/exact_output.rs` 已经有有界的 `extract_single_json`，可拒绝多个 JSON 值并提取唯一 JSON，但结构化输出校验没有复用它。

#### 推荐修复

仅在客户端显式提供 `output_config.format.type=json_schema` 时：

1. 先保留现有“整段裸 JSON”快速路径。
2. 失败后从有界文本中提取唯一一个完整 JSON 候选。
3. 必须再次使用原始客户 Schema 严格校验；缺字段、类型错误、额外字段、多 JSON 值继续拒绝。
4. 返回给客户的是规范化后的 JSON，不返回 Markdown 围栏和说明文字。
5. 第一轮能提取成功时不再调用第二轮；只有第一轮没有唯一合法候选时才做现有的一次恢复。

可修复性：**很高**。按当前 252 个历史样本回放，预期 212 个首轮成功、40 个第二轮成功，最终 252/252 成功。

客户影响：

- 普通聊天、工具调用、未携带 `output_config.format` 的请求完全不变。
- 结构化输出客户会从“收到错误”变为“收到 Schema 合法 JSON”。
- 约 84.1% 的这类样本可省掉第二次上游调用，降低延迟和上游消耗。
- 这是输出归一化，不增加系统提示词；仅保留现有失败时恢复轮的有限提示。

### 5.2 P0：工具 Schema 恢复路径没有实际重试（25 次）

#### 分布

| 错误 | 次数 |
|---|---:|
| `Edit` 缺少 `file_path` | 10 |
| `AskUserQuestion.questions` 应为数组 | 6 |
| `read_file` 缺少 `path` | 6 |
| `AskUserQuestion` 同时存在顶层字段漂移 | 1 |
| `Bash.run_in_background` 不是 boolean | 1 |
| `StructuredOutput` 缺少两个 required 字段 | 1 |

#### 根因判断

所有 25 个 trace 都只有一次上游 attempt。现有 `ProbationBuffer` 在完整工具块出现时会把它视为语义输出并进入 committed 状态；随后 Schema 校验失败时，`ToolAttemptState::should_retry` 因 `semantic_output_started/tool_forwarded` 不满足而拒绝透明重试。因此代码中虽然存在 Schema retry instruction，生产现场没有真正使用第二轮。

#### 推荐修复

1. 在完整工具 JSON 通过 Schema 校验前，不向客户端提交该工具块。
2. 第一轮工具非法、且尚未交付正文或其他工具时，丢弃整轮缓冲，用带有缺失字段/类型约束的工具描述重试一次。
3. 第二轮仍非法时返回明确 `upstream_tool_schema_error`，绝不把非法工具交给客户端执行。
4. 只允许无歧义的本地修复，例如把确实是 JSON 数组文本的字符串解析为数组、把严格的 `"true"/"false"` 转为 boolean；绝不猜 `file_path`、`path` 或业务值。
5. 已经向客户交付正文或合法工具后，不透明重试，避免重复文本或重复执行工具。

可修复性：**中高**。缺少路径无法本地伪造，但可通过一次真正有差异的上游重试恢复；仍受上游模型是否改正影响。

客户影响：

- 合法工具调用的参数本来就必须等到完整 JSON 才能执行，额外可见延迟应很小。
- 只有上游首次生成非法工具时会增加一次重试，带来额外延迟和上游用量。
- 修复后不会执行伪造路径或错误参数，安全性与对话连续性提升。
- 若为了重试而缓冲 thinking，thinking 的实时展示会延后；建议只缓冲“尚未完成的工具块”，不要默认缓冲普通正文。

### 5.3 P1：上游图片格式不支持（26 次）

#### 证据

- 26 次实际只有 4 个不同的请求体形状，是客户端在短时间内重复流式/非流式尝试。
- 全部来自同一个 Key、同一个约 199 条消息的长会话。
- 请求包含 5 张声明为 `image/png` 的图片；最新一张位于当前轮，原始数据仅约 2,388 字节。
- Kiro 返回 `IMAGE_FORMAT_UNSUPPORTED`，说明请求已越过 RS 本地预算预检并到达上游。

#### 推荐修复

在生成 Kiro 请求前对每张图执行有界预检：

1. Base64 解码后用 magic bytes/图片解码器验证实际格式，而不是只相信 `media_type`。
2. 对可解码但 Kiro 不兼容的静态图片，归一化为标准 RGB/RGBA PNG；历史图仍可走现有 JPEG 压缩。
3. 对真正损坏、截断或无法解码的图片，本地返回包含消息索引和图片索引的清晰 400，不把正文或图片内容写入日志。
4. 错误快照只记录格式、字节数、尺寸和哈希摘要。

可修复性：**部分可修**。如果图片像素可被本地解码，标准化后大概率可恢复；如果原始数据本身损坏，只能更快、更清楚地失败，不能凭空修复图片。

客户影响：

- 仅图片请求受影响；文本对话不变。
- PNG 归一化保持像素无损，但会移除元数据、色彩配置或动画信息；模型只需要静态像素，通常是可接受的。
- 增加少量图片解码 CPU 和延迟，同时减少无效上游请求。

### 5.4 P1：120 秒 read timeout 与流 watchdog 竞态（18 次）

#### 证据

- 13 次流式 `stream_read_error`、4 次非流式缓冲 `stream_interrupted`、1 次显式 `stream_idle_timeout`。
- 17 次共同错误为 `error decoding response body`。
- 上游 endpoint 均为 `ide`，HTTP 状态均为 200，分布在多个凭据上，不是单个坏账号。
- 绝大多数总耗时约 121–142 秒；长输出样本也是最后一段数据后约 120 秒中断。
- 13 次流式 read error 中，7 次在没有任何正文/首 token 时中断，6 次已交付部分内容。
- 4 次非流式请求尚未向客户交付正文，因此具备安全重试条件。
- 生产配置 `streamIdleTimeoutSecs=120`；provider 同时把相同值设置为 reqwest read timeout，而应用层又维护独立 120 秒 watchdog。

#### 推荐修复

1. 不让 reqwest read timeout 与应用 watchdog 使用完全相同的截止时间：流式请求可关闭底层 read timeout，或设置为 watchdog + 5~15 秒，让应用层先产生可解释的 `stream_idle_timeout`。
2. 记录 reqwest error chain、`is_timeout`、收到的字节数、最后一帧时间和 decoder 剩余缓冲长度；不要只保留 `error decoding response body`。
3. 对“零字节、未交付正文/工具”的流式请求安全重试一次。
4. 对非流式缓冲请求，只要没有向客户返回任何内容，也可在有界条件下重试一次。
5. 已经交付部分正文/工具的流式请求不能透明重试，否则会造成重复文本或重复工具执行，只能发送一次明确 SSE error 并保留快照。

可修复性：**部分可修**。可消除模糊分类，并可能恢复 7 个零输出流式样本和 4 个非流式样本；已经交付部分内容的中断无法无风险自动续接。

客户影响：

- 仅异常慢或中断的请求可能增加一次上游调用、延迟和用量。
- 普通请求的 1 秒 connected/ping 首响应不需要移除；ping 不是上游正文，仍可与内部重试并存。
- 不建议简单把超时从 120 秒提高到 300 秒，这只会让客户更久后才看到相同失败。

### 5.5 P1：RS 本地图片预算 400（生产 22 次）

这 22 次与 8991 同属 `prepare_outbound_kiro_bodies` 的本地 400。HTTP 400 分支在该位置只对应 `ImageBudgetError::Exceeded`；序列化/内部失败会返回 500，因此可以确定是预算超限。

22 次由 7 个不同请求体组成，其中 5 个请求体各被客户端原样重试 4 次。两类会话形状为：

- 15 条消息、15 张图；
- 86~87 条消息、4 张嵌套在工具结果中的图。

应与 8991 一并使用“普通压缩超限后先尝试激进压缩”的修复。能恢复多少生产样本需要用脱敏图片 fixture 回放，当前快照已删除原始 Base64，不能伪造确定结论。

### 5.6 P2：客户端主动断开（45 次）

- 43 次已经向客户端发送过字节，只有 2 次是零字节断开。
- 平均约 105.9 秒，最长约 300 秒。

这类主要是客户端取消、New API/反向代理超时、网络断线或用户关闭会话，不能直接认定为 RS 故障，也不应计入“上游错误率”。

建议：管理端单独展示“客户端取消/断开”，并按客户端、代理层超时和是否已首字节分组。RS 在 `earlyStreamHandshake=true` 时继续保留 1 秒 ping；不要为了减少该计数而让后台请求在客户端断开后继续消耗上游资源。

### 5.7 P2：上游 error 事件（5 次）

这 5 次均为上游 HTTP 200 后通过事件流返回 error，分散在不同模型和凭据。当前对外和 trace 只保留 `Upstream reported error`，不足以判断是上下文、内容策略、模型容量还是协议问题。

建议先增强脱敏诊断：记录上游 error type、有限长度的安全 message、事件类型、是否已有语义输出和 decoder 状态；只有确认是瞬态且尚未交付内容时才重试。现阶段不能宣称可完全修复。

## 6. 日志与错误快照问题

### 6.1 Docker DEBUG 日志过于嘈杂

全局 DEBUG 会记录大量 `h2::codec` 每帧日志，导致 json-file 很快轮转，真正需要的 12:40~13:04 现场已经不在当前 Docker 日志中。服务器性能强并不能解决“关键信息被噪声挤出保留窗口”的问题。

推荐使用按模块过滤：

- `kiro_rs=debug`
- `h2=info`
- `hyper=info`
- `reqwest=info`

同时在 Docker 配置合理的 `max-size/max-file`。这样仍保留 RS 请求转换、重试、图片预算和协议状态的 DEBUG，而不记录每个 HTTP/2 frame。

客户影响：无协议变化；降低日志锁竞争、CPU 和磁盘写入，通常是正向影响。

### 6.2 本地 400 快照丢失具体原因

当前 `finalize_immediate_response` 只记录 `local response returned HTTP 400 Bad Request`，没有保存响应中的安全错误码和图片预算统计。此次必须结合源码分支才能确认 22 次属于图片预算。

推荐直接把结构化的安全原因交给 tracer：

- `error_type=image_budget_exceeded`
- 图片总数、历史/当前数量
- 压缩前后字节数、预算
- 普通/激进压缩是否执行

不记录 Base64、图片内容或客户正文。

客户影响：仅诊断字段变化，对 API 响应与对话无影响。

### 6.3 快照容量

生产 `error_snapshots.db` 当前约 50.8 GB；配置上限为 200 GB、最少保留 100 GB 空闲、保留期 90 天，并且会捕获已恢复请求体。服务器有足够磁盘时可以继续保留，但管理端必须将 recovered warning 与最终失败分开，查询必须走 `ts_epoch` 索引，避免扫描整库。
