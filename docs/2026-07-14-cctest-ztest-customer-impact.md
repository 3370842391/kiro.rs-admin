# CCTest/Ztest 与错误日志优化：客户影响说明

更新时间：2026-07-14

本文记录本轮协议兼容、工具调用与错误快照修改对客户对话、工具执行、延迟、Token 和计费的影响。状态以代码复审和 8991 验收结果为准；未完成项目不得部署到生产 8990。

## 已完成且风险较低

### 历史 `tool_use.id` 规范化

- 客户传入的合法 ID（1–64 字节，仅 ASCII 字母、数字、`_`、`-`）保持不变。
- 含 `/`、`.`、`:`、空值或超长的历史 ID，只在本次发往 Kiro 的请求副本中映射为稳定安全 ID；对应 `tool_result` 使用同一映射。
- 不修改客户端保存的历史，不修改此前已经返回给客户端的 ID，不增加系统提示词或 Token。
- 客户可感知变化：原本会被 Kiro 立即 400 的 session resume/工具续轮通常可以继续；重复、孤立、顺序错误或映射冲突的损坏历史改为 RS 本地返回可理解的 400。

## 已完成的工具 Schema 契约

目标行为：

- 合法工具调用保持原样。
- 仅对客户端 Schema 中 `required` 的 `const` 或单值 `enum` 做确定性修复；不从 prompt 猜 nonce/城市/业务值，不做字符串转数字，不填空字符串或零值，不删除额外字段来“凑检测”。
- 非固定字段缺失、类型错误、非法额外字段或未声明工具不会交付客户端执行。
- 首轮尚未交付正文或工具时，Schema 错误最多触发一次上游重试；已有正文后不透明重试，避免重复正文或重复工具。

客户影响：

- 对话内容：普通文本与没有工具的请求不受影响。
- 工具参数：固定契约字段可能被修正为客户端明确声明的唯一值；其他业务参数绝不猜测或强制改写。
- 延迟：只有上游首次生成非法工具参数、且尚未向客户端提交任何语义内容时，可能增加一次上游调用和等待时间。
- 工具执行：非法参数不再下发执行，客户端会收到协议错误；这比执行错误工具更安全，但依赖过去宽松行为的客户端可能感知失败率变化。
- 多工具流：在无法完整缓冲所有工具的路径中，前一个合法工具可能已交付，后一个非法工具再使流终止；客户端必须按工具 ID 做幂等处理。此项在 8991 验收前仍视为待确认风险。
- Token/计费：不向工具描述复制 Schema，不增加系统提示词；仅发生安全重试时会产生额外上游调用。本轮不修改缓存命中或用量拆分口径。

### 固定字段修复后的再次去重

- Schema 的 `const`/单值 `enum` 修复完成后，会重新比较原生结构化工具调用与从正文 `<invoke>` 捞回的工具调用。
- 若两者在修复后变成相同的“工具名 + 参数”，只保留原生结构化调用，防止客户端把同一工具执行两次。
- 两个原生结构化调用即使参数相同也不会被合并，避免破坏客户明确请求的并行调用语义。
- 客户影响：消除重复工具执行及其副作用；仅会移除与原生调用完全等价的文本捞回副本，不改变普通正文、工具参数和原生工具 ID。

### Mixed WebSearch 前置校验

- 最终轮同时包含 `web_search` 与客户工具时，RS 会先完成全部客户工具的 Schema 校验、确定性固定值修复和修复后去重，再调用 MCP 搜索。
- 任一客户工具非法时，MCP 搜索调用数为 0，并返回明确的 `upstream_tool_schema_error`；不会先产生搜索副作用再报告工具错误。
- 客户影响：非法 mixed-tools 请求会更早失败且不会产生无用搜索；合法请求仍只执行原有搜索，只有本地校验的微小 CPU 开销，不增加 Token 或上游调用次数。

## 已完成的错误快照日志

- 只在失败、中断、协议错误或配置允许的“重试后恢复”请求上保存诊断快照；正常单跳成功请求不写大正文。
- 会保存脱敏后的客户请求、工具 JSON/Schema、Kiro 请求、上游响应、重试链和流尾部，以便按 trace ID 复现问题。
- 永不保存 Authorization、API Key、access/refresh/id token、client secret、Cookie、密码或凭据字段；图片、PDF、data URI 和长 base64 只保存长度与 SHA-256。
- 快照写入、压缩或数据库故障不得改变客户 HTTP 状态、SSE 事件或错误正文。
- 客户影响：失败路径会有少量脱敏、压缩和磁盘写入开销；正常成功请求不承担大 BLOB 写入。不会改变 Token、缓存计费或模型正文。
- 存储策略：独立 `error_snapshots.db`，默认保留 90 天、上限 200GB、至少保留 100GB 空闲；critical 和手动 pin 不参加自动清理。上述值后续可由管理端配置。
- 管理端已经提供列表、筛选、详情、下载、固定、删除、清理和容量治理入口；这些管理操作不进入客户对话路径。

## 后续协议兼容项的预期影响

| 项目 | 客户可感知变化 | Token/计费 |
|---|---|---|
| `output_config.format` | 带 JSON Schema 的流式请求会完整缓冲后再返回，首字延迟增加，但不再交付半截/非法 JSON | 仅非法输出的一次安全重试可能增加上游消耗 |
| 严格 SSE 顺序 | 移除 `message_start` 之前的非标准 ping；慢上游首字可能稍晚，官方 SDK 兼容性更好 | 无变化 |
| thinking signature | 原生签名优先透传；无原生签名只生成 RS 自有回放令牌，不冒充 Anthropic 官方签名 | 无变化 |
| 原生 WebSearch | 中间 block、usage、stop reason 会变化；最终答案与搜索结果仍保留 | 不修改计费口径 |
| 文本型 PDF | 文本 PDF 的唯一标识提取更稳定；扫描 PDF 仍明确不支持 OCR | 不增加系统提示词 |
| 协议指纹收口 | 只修有 fixture/官方基线证明的差异，不随机改 ID/Header | 无变化 |

## 明确不实施

- 不硬编码检测站 nonce、城市、天气值、报告 UUID 或答案。
- 不从用户文本猜工具参数。
- 不用固定占位符冒充 Anthropic 私钥签名。
- 不为了分数重新注入大型 system prompt。
- 不修改已经通过审计的缓存计费与 Token 拆分口径。
- 本轮只允许部署到独立 8991 验收；生产 8990 不在实施范围内。
